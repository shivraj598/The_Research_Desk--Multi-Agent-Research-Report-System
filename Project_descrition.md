## 1. Project Overview

The Multi-Agent Research & Report System is an autonomous research pipeline built from four specialized AI agents that collaborate through a shared state graph to produce a structured, fact-checked report on any given topic — with a human approval step before final delivery.

Instead of relying on a single LLM call (which tends to produce shallow, unverified, and poorly organized output), this system decomposes research the way a real team would: one agent plans, one gathers, one analyzes, one writes, and one verifies. Each agent has a narrow, well-defined responsibility, and LangGraph manages the handoffs, shared memory, and control flow between them.

### Problem It Solves

Manual research for a comprehensive report typically takes 4–6 hours: identifying subtopics, searching, reading sources, cross-referencing facts, and writing coherently. A single LLM prompt can approximate this but usually skips depth, hallucinates details, and can't verify its own claims. This system replicates the output quality of a 4-person research team in under 10 minutes.

**Target users:** consultants, analysts, journalists, startup founders — anyone who needs fast, credible, well-organized research without doing the legwork themselves.

## 2. System Architecture

### 2.1 Agent Roles

|Agent|Responsibility|
|---|---|
|**Orchestrator**|Breaks the user's topic into 4–6 discrete research subtasks; decides task order and routes work to the Research Agent; tracks overall progress in shared state|
|**Research Agent**|For each subtask, queries Tavily Search API and scrapes/fetches source content in parallel; deduplicates and stores raw source material|
|**Analyst Agent**|Reads raw scraped content per subtask and extracts structured insights — key facts, figures, quotes, and their source attributions|
|**Writer Agent**|Compiles all structured insights across subtasks into a single, coherent, professionally formatted report with sections, headings, and a summary|
|**Fact-Checker Agent**|Re-checks every claim in the draft report against the original sourced content, flags unsupported or contradicted claims, and either corrects or annotates them|

### 2.2 Workflow (High Level)

```
User Topic
    │
    ▼
[Orchestrator] ── splits topic into 4–6 subtasks
    │
    ▼
[Research Agent] ── parallel Tavily search + scrape per subtask
    │
    ▼
[Analyst Agent] ── extracts structured insights per subtask
    │
    ▼
[Writer Agent] ── compiles insights into full report draft
    │
    ▼
[Fact-Checker Agent] ── validates every claim against sources
    │
    ▼
[Human Approval Gate] ── user reviews, approves or requests revision
    │
    ▼
Final Report Delivered
```

### 2.3 Shared State

All agents read from and write to a single shared state object (a LangGraph `StateGraph` built around an `AppState` dataclass). This is what makes the system a true multi-agent graph rather than a chain of isolated prompts — every agent sees the accumulated context of everyone who acted before it.

**Representative `AppState` fields:**

- `topic: str` — original user-provided research topic
- `subtasks: list[str]` — subtasks generated by the Orchestrator
- `raw_sources: dict[str, list[dict]]` — scraped content per subtask, keyed by subtask
- `insights: dict[str, list[dict]]` — structured extracted insights per subtask
- `draft_report: str` — compiled report from the Writer Agent
- `fact_check_results: list[dict]` — flagged/corrected claims from the Fact-Checker
- `final_report: str` — approved, final version
- `approval_status: str` — pending / approved / revision_requested

### 2.4 Human-in-the-Loop Approval Gate

Before the report is finalized, the workflow pauses at a LangGraph interrupt point. The user reviews the fact-checked draft and either:

- **Approves** → the graph proceeds to output the final report, or
- **Requests revision** → the graph routes back to the Writer Agent (or further back to the Analyst Agent, depending on the nature of the feedback) with the user's notes added to shared state.

This gate is intentionally the last piece to build — the automated end-to-end flow should work reliably first.

## 4. Build Plan

Recommended build order, from simplest working version to full system:

1. **Start with 2 agents: Researcher + Writer.** Get a minimal end-to-end flow working — topic in, rough report out — before adding complexity.
    
2. **Define the shared state first.** Use LangGraph's `StateGraph` with a shared `AppState` dataclass that every agent reads from and writes to. Getting this right early avoids painful refactors later.
    
3. **Add the Analyst Agent.** Insert it between Research and Writer so raw scraped content gets structured into clean insights before the Writer ever sees it.
    
4. **Add the Fact-Checker Agent.** Insert it after the Writer, checking the draft report's claims against the original sourced content stored in shared state.
    
5. **Add the human approval gate last.** Build and stabilize the fully automated flow first, then insert the LangGraph interrupt for human review as the final step before delivery.
    
6. **Wrap it in FastAPI.** Expose a single endpoint (e.g., `POST /research`) that accepts a topic and returns the report (or a job ID for async polling), so the whole system can be demoed live with one API call.
